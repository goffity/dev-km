## Context: Before

- **Problem**: Orchestrator injected static TDD instructions into specialist via `inject_tdd_instructions()` function (~66 lines) in `specialist-spawn.sh`. This was rigid, verbose, and duplicated logic that TDG plugin already handles.
- **Existing Behavior**: `spawn_with_task()` had a `tdd_enabled` parameter (auto/true/false) that auto-detected TDG.md and injected hardcoded TDD rules into `.claude/rules/specialist.md`.
- **Why Change**: TDG plugin (`tdg@tdg-marketplace`) is globally available to all Claude instances. Specialists should use `Skill("tdg:tdg")` directly instead of receiving static injected instructions.
- **Metrics**: `specialist-spawn.sh` was ~600 lines with complex TDD injection logic.

---

## Context: After

- **Solution**: Removed `inject_tdd_instructions()` entirely. Added a "TDD WORKFLOW (Autonomous)" section to `CLAUDE.specialist.md` template that teaches specialists when/how to use TDG plugin on their own.
- **New Behavior**: Specialist autonomously decides whether to use TDD based on task type, checks for TDG.md, tries `Skill("tdg:tdg")` first, falls back to manual Red-Green-Refactor.
- **Improvements**: -101 lines removed, +10 lines added (validation). Cleaner separation of concerns. Specialist is smarter about TDD decisions.
- **Metrics**: `specialist-spawn.sh` reduced to ~534 lines. No more rigid orchestrator-driven TDD injection.

## Lessons Learned

- **TDG plugin is globally available**: No need for orchestrator to inject TDD rules when the plugin is already in `~/.claude/settings.json`
- **GitHub API for PR comment replies**: Use `gh api repos/{owner}/{repo}/pulls/{pr}/comments` with `-F in_reply_to={comment_id}` (must be `-F` not `-f` for numeric field)
- **`gh issue create` returns URL**: Parse issue number from URL output, don't try `--json` flag
- **Squash merge causes local divergence**: After squash merge via `gh pr merge --squash`, local branch diverges from remote - use `git reset --hard origin/main` to sync
