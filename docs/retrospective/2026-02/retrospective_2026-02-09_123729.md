## Context: Before

- **Problem**: `cmd_create()` did not support `--due` flag, while `cmd_create_subtask()` and template-based commands already had it
- **Existing Behavior**: Users had to use the Jira UI or API directly to set due dates on created issues
- **Why Change**: Feature parity - all create commands should support `--due` for consistency
- **Metrics**: 4 of 5 create commands supported `--due`, but only 1 (`cmd_create_subtask`) had validation

---

## Context: After

- **Solution**: Added `--due YYYY-MM-DD` flag to `cmd_create()` and extracted `validate_due_date()` helper
- **New Behavior**: `jira create PROJECT "Summary" --due 2026-03-15` sets due date on creation
- **Improvements**: All 5 create commands now consistently validate date format via shared helper
- **Metrics**: 5/5 commands support `--due` with validation, 0 duplicated validation blocks (was 5)

## Decisions & Rationale

| Decision | Options Considered | Chosen | Rationale |
|----------|-------------------|--------|-----------|
| Validation approach | Inline regex in each function vs shared helper | Shared `validate_due_date()` helper | Reduces duplication from 5 copies to 1, follows `validate_issue_key()` pattern |
| Validation strictness | Syntactic only (YYYY-MM-DD) vs semantic (valid calendar date) | Syntactic only | Jira API rejects invalid dates server-side; regex is sufficient for format check |
| Payload construction | Separate payload for --due vs conditional jq pipe | Conditional jq pipe | Consistent with `--labels` and `--points` pattern: `if $field != "" then .fields.key = $value else . end` |
