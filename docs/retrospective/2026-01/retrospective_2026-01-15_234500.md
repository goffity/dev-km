## Context: Before

- **Problem**: GraphQL queries for review threads limited to 100 items
- **Existing Behavior**: Single query without pagination
- **Why Change**: PRs with >100 review threads would miss comments
- **Metrics**: Max 100 threads per query

---

## Context: After

- **Solution**: Cursor-based pagination with `pageInfo { hasNextPage endCursor }`
- **New Behavior**: Loop through all pages until `hasNextPage == false`
- **Improvements**: Can handle unlimited review threads
- **Metrics**: All threads processed regardless of count

## Decisions & Rationale

| Decision | Options Considered | Chosen | Rationale |
|----------|-------------------|--------|-----------|
| Pagination approach | Offset vs Cursor | Cursor-based | GitHub API standard, more efficient |
| Page size | 50, 100 | 100 | Max allowed by GitHub |
| Error handling | Fail fast vs Continue | Continue with current count | Resilient to partial failures |

### Key Code Pattern

```bash
while [[ "$has_next_page" == "true" ]]; do
    # Query with cursor
    result=$(gh api graphql ... -f cursor="$cursor")
    
    # Process results
    page_count=$(echo "$result" | jq '...')
    total_count=$((total_count + page_count))
    
    # Get next page info
    has_next_page=$(echo "$result" | jq -r '.pageInfo.hasNextPage')
    cursor=$(echo "$result" | jq -r '.pageInfo.endCursor')
    
    # Safety break
    if [[ -z "$cursor" || "$cursor" == "null" ]]; then
        break
    fi
done
```
